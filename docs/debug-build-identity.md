# Debug vs. release build identity

Debug builds carry a **distinct application identity** from release/profile
builds on every platform, so a debug build and a "real" (release/profile) build
can be installed and run side by side — the release build can be dogfooded on
real data while a debug build is driven for development or visual verification,
with no collision.

The rule is uniform: the debug identity is the release identity plus a `.debug`
suffix (or, where there is no OS-level identity, a `_DEBUG` class suffix). The
visible window title / launcher label stays **`BrainFrame`** in every mode — the
intended visual tell between debug and release is the **dev icon** (see
"Debug icon seams" below), not a renamed title.

## Where the identity is set

| Platform | Debug identity | Set in |
| --- | --- | --- |
| Linux (and Pi) | `tech.brainframe.app.debug` (GApplication ID + `WM_CLASS`) | `linux/CMakeLists.txt` — `.debug` appended when `CMAKE_BUILD_TYPE` is `Debug` |
| Android | `tech.brainframe.app.debug` (applicationId) | `android/app/build.gradle.kts` — `applicationIdSuffix = ".debug"` on the `debug` build type |
| iOS | `tech.brainframe.app.debug` (bundle ID) | `ios/Runner.xcodeproj/project.pbxproj` — `PRODUCT_BUNDLE_IDENTIFIER` in the Debug config |
| macOS | `tech.brainframe.app.debug` (bundle ID) | `macos/Runner.xcodeproj/project.pbxproj` — `PRODUCT_BUNDLE_IDENTIFIER` override in the Runner Debug config (base value lives in `Configs/AppInfo.xcconfig`) |
| Windows | window class `FLUTTER_RUNNER_WIN32_WINDOW_DEBUG` | `windows/runner/win32_window.cpp` — `#ifdef _DEBUG` (Windows has no OS-level app ID; the window class is the closest analog and lets external tooling target the debug window) |

Profile builds share the **release** identity on every platform.

The Linux visual-verification tool (`tool/appshot.sh`) relies on this: it
targets only the `tech.brainframe.app.debug` window, so a release build left
running on real notes is never captured, clicked, resized, or quit. See
`.claude/rules/visual-verification.md`.

## Debug "dev" icon

Debug builds carry a distinct **dev icon** (a second visual tell alongside the
`.debug` identity). The source is `brainframe-dev.png` at the repo root — the
debug counterpart of `brainframe.png`, from which the release icons are
generated by `flutter_launcher_icons`.

`flutter_launcher_icons` only supports flavors for Android and iOS (not macOS
or Windows), so the debug icon is generated for **every** platform by one
script instead:

```sh
python3 tool/gen_debug_icons.py    # requires Pillow
```

It mirrors the exact asset layout the release icons use:

| Platform | Debug icon asset | How the debug build selects it |
| --- | --- | --- |
| Android | `android/app/src/debug/res/` (foreground drawable, legacy bitmap, adaptive XML, background color) | Gradle resource merging overrides `main` for the `debug` build type |
| iOS | `ios/Runner/Assets.xcassets/AppIcon-Debug.appiconset/` | `ASSETCATALOG_COMPILER_APPICON_NAME = AppIcon-Debug` in the Debug config |
| macOS | `macos/Runner/Assets.xcassets/AppIcon-Debug.appiconset/` | `ASSETCATALOG_COMPILER_APPICON_NAME = AppIcon-Debug` in the Runner Debug config |
| Windows | `windows/runner/resources/app_icon_debug.ico` | `#ifdef _DEBUG` guard on `IDI_APP_ICON` in `windows/runner/Runner.rc` |

The image assets are regenerated by the script and safe to re-run; the
build-config wiring (the `ASSETCATALOG_COMPILER_APPICON_NAME` values and the
`Runner.rc` guard) is a one-time committed change, not regenerated.

### Android adaptive-icon note

The dev artwork carries its "dev" badge in a **corner**. Android launchers crop
the outer ~17% of an adaptive icon and then mask it to a circle/squircle, so a
corner element is easily lost — and *filling* the tile makes it worse (the badge
is cropped away entirely). The debug adaptive icon therefore insets the whole
square into the launcher safe zone (`ic_launcher.xml`, 25%) and paints the
adaptive background (`ic_launcher_background`) with the artwork's own dark tone,
so the full square — badge included — survives every mask and still reads as one
cohesive dark icon. Desktop/Windows show the artwork square and unmasked, so
they need none of this. If the badge is ever moved out of the corner, the inset
can drop back toward the release icon's 16%.

**Linux** has no in-toolchain launcher-icon mechanism (the release config skips
it too); when `.desktop`/window-icon packaging is added, key the debug icon off
`CMAKE_BUILD_TYPE` the same way the app ID is.
