# Internationalization & localization design

- **Status:** accepted (2026-07-04)
- **Author:** Claude
- **Date:** 2026-07-04

## TL;DR

BrainFrame should route every user-facing string through a localization
layer now, while there are only ~26 of them, and **enforce** that routing so a
hardcoded string fails fast forever after. This document picks the mechanism
(Flutter's official `gen-l10n` over ARB files), the seed locales (English plus
a *generated* `en_XA` pseudo-locale), the enforcement (a cheap cross-platform
Dart check plus an in-IDE `custom_lint` rule), the small set of
conventions — key naming, generated-file handling, the future
language-switcher seam — that keep the discipline honest as the UI grows, and
a beginner-oriented guide so a non-expert contributor can add a language.
Localization is **two surfaces**: the UI strings above, and the bundled
tutorial/help engrams, which are whole markdown documents — this doc covers
both, giving the built-in content a locale-partitioned asset layout with
per-file fallback to English so a lagging translation degrades gracefully.

Nothing is localized in the code today: the three UI files
([engram_browser.dart](../../lib/engram/ui/engram_browser.dart),
[engram_switcher.dart](../../lib/engram/ui/engram_switcher.dart),
[help_overlay.dart](../../lib/engram/ui/help_overlay.dart)) hold literal
strings, and [app.dart](../../lib/app.dart) hardcodes the title. This is a
clean retrofit at the cheapest possible moment, not a migration of an
entrenched pattern.

## Why now, and why enforced

The cost of i18n is roughly linear in the number of strings already written
against it. At 26 strings the migration is an afternoon; at 2,600 it is a
quarter and never fully finished. The same asymmetry applies to *discipline*:
a lone hardcoded string is trivial to fix the day it lands and nearly
impossible to find a year later. So the deliverable is not just "wire up
`gen-l10n" — it is "make the next hardcoded string a build failure." Both
halves ship together.

## Vocabulary

| Term | Meaning |
| --- | --- |
| ARB | *Application Resource Bundle* — the JSON dialect Flutter's `gen-l10n` reads. One file per locale (`app_en.arb`). |
| Template ARB | The base-locale file (`app_en.arb`) that defines the keys and carries `@`-metadata descriptions. |
| `AppLocalizations` | The class `gen-l10n` generates from the ARBs; screens read strings off `AppLocalizations.of(context)`. |
| Pseudo-locale | A machine-generated fake locale (`en_XA`) that accents and pads every string, used to surface un-localized or overflow-prone text. |
| Widget text slot | A widget parameter that renders user-visible text: `Text('…')`, `label:`, `labelText:`, `hintText:`, `tooltip:`, `semanticLabel:`, `helpText:`, dialog `title:`, etc. |

## Mechanism: official `gen-l10n` over ARB

Flutter ships first-party localization: `flutter_localizations` (the SDK
package that localizes Material/Cupertino widgets themselves) plus the
`gen-l10n` build step that turns ARB files into a typed `AppLocalizations`
class. We adopt it wholesale.

Why the built-in path over a third-party package (e.g. `slang`):

- **Zero third-party runtime dependency**, in a project whose ethos is
  standard conventions and minimal surface.
- **It is the seam `MaterialLocalizations` already uses** — date pickers,
  the "OK/Cancel" of platform dialogs, text-selection handles, and semantics
  come localized for free once `localizationsDelegates` is wired, and stay in
  step with the adaptive Material+Cupertino facade
  ([[ui-adaptive-first]]).
- **Typed access, no BuildContext-free magic to learn.** Keys become getters;
  a missing key is a compile error, not a runtime null.

The DX niceties `slang` offers (context-free access, compile-time key trees)
do not outweigh adding a codegen dependency and diverging from the path every
Flutter contributor already knows.

### Wiring

`MaterialApp` in [app.dart](../../lib/app.dart) gains the two delegates and
the supported-locale list, and trades its hardcoded `title:` for a localized
`onGenerateTitle:`:

```dart
MaterialApp(
  onGenerateTitle: (context) => AppLocalizations.of(context).appTitle,
  localizationsDelegates: AppLocalizations.localizationsDelegates,
  supportedLocales: supportedLocales, // en always; en_XA only in debug/profile
  // …existing theme / home wiring unchanged…
)
```

`supportedLocales` here is the generated `AppLocalizations.supportedLocales`
with the pseudo-locale filtered out of **release** builds (see Seed locales),
so real users never see `en_XA` in device-locale resolution while developers
and tests still can.

Screens then read strings the same way they already read settings and the
active engram — off an `.of(context)` — so the pattern is familiar:

```dart
final l10n = AppLocalizations.of(context);
// Text('New engram')  ->  Text(l10n.switcherNewEngram)
```

## Conventions (the parts that need consensus)

### Key naming

`lowerCamelCase`, prefixed by the screen/area when it disambiguates:
`appTitle`, `switcherNewEngram`, `switcherOpenFolder`, `browserUnreadable`,
`helpTitle`, `helpIndex`. Keys name the *role*, never the English wording, so a
copy change never forces a key rename. Every key in the template ARB carries an
`@key` block with a one-line `description`; placeholders and plurals use ICU
syntax (`"{count, plural, =0{…} one{…} other{…}}"`) with typed `placeholders`
metadata. There are no placeholders in the current 26 strings, but the
convention is fixed now so the first one that needs it has a pattern.

### Generated files

Target Flutter is 3.44 (the synthetic-package l10n mode is gone), so
`gen-l10n` emits real source. `l10n.yaml`:

```yaml
arb-dir: lib/l10n
template-arb-file: app_en.arb
output-dir: lib/l10n/gen
output-localization-file: app_localizations.dart
output-class: AppLocalizations
nullable-getter: false
```

Generated output under `lib/l10n/gen/` is **git-ignored** and regenerated by
the l10n build step (`flutter: generate: true` runs `gen-l10n` on
`pub get` / `test` / `build`), matching how the repo already treats derived
artifacts. It is **excluded from the coverage trace** — the generated
per-locale lookup tables are not code we test — by adding `lib/l10n/gen/` to
the single `EXCLUSIONS` list in [tool/coverage.sh](../../tool/coverage.sh)
(and its mirror in `.github/workflows/coverage.yml`), exactly as `.g.dart` is
excluded today. The alternative — committing the generated class — is
rejected: it is derived code that would churn on every string change and
invite hand-edits.

### Seed locales

`en` (base) and `en_XA` (pseudo). English is the source of truth; **the pseudo
locale is generated, never hand-written**, by `tool/gen_pseudo_arb.dart`
(mirrors the existing `tool/` scripts). It reads `app_en.arb` and emits
`app_en_XA.arb`, transforming each value by: mapping letters to accented
look-alikes (`New engram → Ñéw éñgrám`), padding ~35% to expose layouts that
only fit English, and bracketing (`[…]`) so truncation is visible — while
leaving `{placeholders}` and ICU keywords untouched. Because it is generated,
`en_XA` is always in perfect key-sync with the base; the enforcement gate
fails if it drifts (regenerate to fix), the same freshness contract the
Markdown gate uses.

`en_XA` is a **debug/profile-only** locale: it exists so tests and developers
can select the pseudo-locale, but it is filtered out of `supportedLocales` in
**release** builds (`kReleaseMode`) so it never appears to real users during
device-locale resolution. It needs no human translator. The `app_en_XA.arb`
file itself always ships in the repo (it is generated and cheap); only its
*visibility as a selectable locale* is gated to non-release builds.

No real second language ships yet. Adding one later is dropping in
`app_<lang>.arb` — the machinery, tests, and enforcement already prove the
path end to end via the pseudo-locale.

### Adding a language is a one-file drop (and a contributor guide)

The point of i18n is that *other people* add languages, so the barrier to
doing so must be low and written down. It genuinely is low: `gen-l10n`
discovers every `app_<locale>.arb` in `lib/l10n/` and folds it into
`AppLocalizations.supportedLocales` automatically, so **adding a language
touches no Dart code** — a translator copies `app_en.arb` to
`app_<locale>.arb`, translates the *values* (leaving keys, `{placeholders}`,
and ICU keywords untouched), drops the file in, and runs the app. The
key-parity test and the enforcement gate then report any key they missed, so
"is my translation complete?" has an automated answer rather than a guessing
game.

Because neither the maintainer nor most contributors are i18n experts, this
path is captured in a **beginner-oriented contributor guide**, shipped as a
deliverable of this work and co-located with the files it describes at
`lib/l10n/README.md` (discovered the moment someone opens the folder to add a
translation). It assumes no prior ARB or `gen-l10n` knowledge, and covers
**both translation tracks**:

- **UI strings** — copy `app_en.arb` to `app_<locale>.arb`, translate the
  *values* only (leave keys, `{placeholders}`, and ICU keywords like
  `plural`/`select` untouched), drop it in; locale-code naming (`app_es.arb`,
  `app_pt_BR.arb`); how to preview locally; and how to read the parity test's
  "missing-key" output.
- **Built-in content** — copy the base-locale pages under
  `assets/engrams/<engram>/en/` into a sibling `<locale>/` directory and
  translate the markdown (a *partial* translation is fine — untranslated pages
  fall back to English per the section above); register the new asset
  directory in `pubspec.yaml` (Flutter's asset listing is not recursive).

A pointer to the guide goes in the top-level `README.md`.

### Locale resolution and the switcher seam

For now BrainFrame follows the **device locale** (Flutter's default
resolution against `supportedLocales`); with only `en`/`en_XA` supported,
users see English unless their device is set to the pseudo-locale. The
forward-looking seam: a future in-app language override lives in
[AppSettings](../../lib/theme/app_settings.dart) (a `localeOverride`
alongside `themeMode`/`designOverride`) and feeds `MaterialApp.locale`, so
adding a language picker later rebuilds only content, never the app root —
the same themeability seam the app already uses. Out of scope to build now;
called out so the settings object is not designed to preclude it.

### Accessibility tie-in

Per [.claude/rules/accessibility.md](../../.claude/rules/accessibility.md),
`Semantics(label:)`, `tooltip:`, and every other spoken/hinted string is
user-facing and **must be localized too** — screen-reader users are exactly
who a hardcoded string strands. `semanticLabel:`/`tooltip:` are therefore
first-class widget text slots in both the migration and the enforcement gate,
not an afterthought.

## Enforcement (both layers)

A hardcoded string must fail fast. Two layers, deliberately redundant because
they catch different moments and have different bypass profiles:

1. **CLI gate — the CI backstop (lands first).** `tool/check_l10n.dart`, run
   via `dart run tool/check_l10n.dart` from a `pre-commit` hook and the CI
   workflow, scans `lib/` (excluding `lib/l10n/` and the generated dir) for
   string literals sitting in widget text slots and exits non-zero if any are
   found. It is written in **Dart, not shell** — so it runs identically on
   Windows, macOS, and Linux with no `grep`/`bash` dependency (`dart` is
   already required to work on the repo), and it parses the analyzer AST rather
   than regex-matching text, so multi-line calls, adjacent-string
   concatenation, and interpolation are handled correctly. Crucially it does
   **not** depend on a contributor having the `custom_lint` plugin loaded, so
   it cannot be silently bypassed, and it also asserts `app_en_XA.arb` is fresh
   against the generator. It slots into the existing `pre-commit`
   (`flutter analyze`/`markdownlint`) + CI infrastructure the same way
   `tool/coverage.sh` does — and, unlike that bash script, carries no shell
   assumption onto the every-commit path.
2. **`custom_lint` rule — the edit-time layer (lands second).** A local lint
   package exposing a `no_raw_widget_strings` rule that flags literals in the
   same slots, surfaced as an IDE squiggle and via `flutter analyze` /
   `dart run custom_lint`. This catches the string *as it is typed*, before a
   commit is ever attempted — the fast feedback the CLI gate can't give.

The CLI gate is the source of truth for "did enforcement pass"; the lint is
the ergonomic front end. Both reference **one** definition of a "widget text
slot" — a single small Dart module the gate and the lint rule share (exact
layout settled in the impl plan; a test keeps them in sync if they can't
import a common file) — so the two layers can never disagree about what
counts. Genuinely non-UI strings (asset paths, map keys,
`throw` messages, debug `print`s) are outside the slot list, so they are not
false-positived; a rare legitimate literal in a slot is opted out with a
narrowly-scoped `// ignore:`/allowlist entry, reviewed like any suppression.

## Localizing built-in content (tutorial & help)

UI strings are only half the surface. The two bundled read-only engrams — the
**tutorial** and **help** ([engram-storage](engram-storage.md) Decision 5) —
are whole markdown *documents* served from the asset bundle by
`AssetEngramStore` at a fixed prefix (`assets/engrams/tutorial/`,
`assets/engrams/help/`), and they are as English-only today as the widgets
are. `gen-l10n`/ARB does not touch them: this is **content** localization — a
different mechanism, and a far larger volume than UI strings. The move mirrors
the engram-storage doc's philosophy: design for it now, while there are four
markdown files, and ship the minimum.

**Locale-partitioned asset layout.** Content moves under a per-locale
subdirectory, with the base locale (`en`) defining the canonical page set:

```text
assets/engrams/tutorial/
  en/ welcome.md   notes/first-note.md   ← base: defines the pages
  es/ welcome.md                         ← partial: fallback fills the rest
assets/engrams/help/
  en/ index.md   markdown-syntax.md
```

**Per-file fallback to base.** `AssetEngramStore` becomes locale-aware: an
engram's page *list* is the base-locale (`en`) file set, and each read
resolves the active locale first, falling back to `en` per file
(`es_MX` → `es` → `en`). A half-translated tutorial shows the translation
where it exists and English elsewhere — a lagging or partial translation
degrades into a mixed-but-whole document, never a broken or empty one. This
graceful degradation matters far more for content than for UI strings, because
content translations are large and will always trail the base.

**Where the locale comes from.** Unlike a filesystem engram, a built-in
engram's content depends on the active locale — known only inside the widget
tree, and changeable at runtime — whereas the repository is built in `main()`
before the app runs. So a built-in engram's store is constructed with the
locale resolved from context at the point the engram is opened (its switcher
target or the help overlay), and a locale change reconstructs the store and
reloads: the same "swap the value, rebuild the subtree" shape `EngramScope`
already uses for switching engrams. The `EngramStore` contract is **not**
widened with a `locale` parameter — filesystem engrams have no locale — so
locale lives in *how the built-in's store is built*, not in the content
interface. The exact injection point is an impl-plan detail; the fixed
decisions are the layout, the base-defined file set, and per-file `en`
fallback.

**Scope now vs. later.** The layout, the locale-aware store, and the fallback
(with tests) land now, shipping **English content only** — the machinery
exists and is proven, but no translated prose is written yet. Actual
translated tutorial/help documents are ongoing content work, contributed the
same way UI translations are and documented alongside them in the same guide.
This capability extends engram-storage Decision 5, which assumed a single
fixed asset prefix per built-in; the Step 6 PR adds a one-line amendment
pointer to that doc's Decision 5 → here (its own amendments are how it has
evolved before), so the accepted doc stays honest without a rewrite.

**Per-file vs. whole-content is a reversible policy knob, not architecture.**
Both models share this exact layout and the `EngramStore` contract; they
differ only in the resolution rule *inside* `AssetEngramStore`. Switching to
all-or-nothing later is a few lines plus tests in one class — no asset move,
no contract change, no content migration — so per-file is chosen for its
immediate payoff (the first translated page is useful at once) with the
stricter model, or a hybrid "complete-locale" flag, available cheaply if
mixed-language navigation ever needs tightening.

## What ships, in order

The design is realized as a design doc (this file, to `accepted`), an impl
plan (`docs/plans/i18n-impl.md`), and numbered PRs, each its own worktree per
[.claude/rules/git-workflow.md](../../.claude/rules/git-workflow.md):

1. **Scaffold** — `flutter_localizations` + `intl` deps, `generate: true`,
   `l10n.yaml`, empty-ish `app_en.arb` (just `appTitle`), and the `MaterialApp`
   wiring. App still runs; nothing user-visible changes.
2. **Migrate** — all 26 strings into `app_en.arb` with descriptions; the three
   UI files (and their `Semantics`/`tooltip` slots) read from
   `AppLocalizations`; widget tests assert localized text renders.
3. **Pseudo-locale + parity** — `tool/gen_pseudo_arb.dart`, `app_en_XA.arb`,
   a key-parity test across ARBs, and a pump-under-`en_XA` test that asserts
   the layer isn't bypassed at runtime.
4. **CLI gate** — `tool/check_l10n.dart` (Dart, cross-platform) +
   `pre-commit`/CI wiring; proven by a deliberately re-hardcoded string
   failing it.
5. **`custom_lint` rule** — the `no_raw_widget_strings` package + analyzer
   wiring.
6. **Built-in content localization** — move the tutorial/help assets under
   `en/` subdirs, make `AssetEngramStore` locale-aware with per-file `en`
   fallback, wire the active locale from context, and add fallback/resolution
   tests. Ships English content only; no translated prose yet. **Also amends
   [engram-storage.md](engram-storage.md)**: add a pointer on its Decision 5
   and its status line, e.g. *"(Amended 2026-07-04 — built-in content is now
   locale-partitioned under `<locale>/` subdirs; see i18n.md Decision 7.)"* —
   the same amendment style that doc already uses, landing with the code that
   makes it true.
7. **Contributor guide** — `lib/l10n/README.md` covering **both** tracks (UI
   strings and built-in content) plus a pointer from the top-level `README.md`.
   Lands last so the guide describes the real, working paths — and is validated
   by actually following it (see Verification).

## Verification

- `flutter analyze` clean with the lint active; `flutter test` green including
  new ARB-parity, localized-text, and pseudo-locale tests; honest coverage
  stays ≥90% ([testing-coverage-standard]).
- `dart run tool/check_l10n.dart` passes on the migrated tree and **fails** on
  a deliberately re-hardcoded string, on Windows/macOS/Linux alike.
- App run with the device locale forced to `en_XA` shows every string in the
  browser, switcher, and help overlay pseudo-localized — no bare English.
- Built-in content resolution is tested: with only `en` pages present, a
  non-`en` locale still serves the full document via fallback; when a
  `<locale>/` page exists it is served instead; and the page *list* matches the
  base-locale set regardless of active locale.
- The contributor guide is validated by *following it* — for UI strings, add a
  throwaway `app_xx.arb`; for content, add a throwaway `assets/engrams/help/xx/`
  page — confirm each is picked up with no Dart change, then remove them. If a
  step is wrong or missing, the guide is wrong, not the reader.

## Decisions

1. **Official `gen-l10n` over ARB, not a third-party package** (2026-07-04).
   First-party, zero runtime deps, and shares the seam `MaterialLocalizations`
   already uses. `slang`'s ergonomics don't justify diverging from the
   standard Flutter path in a conventions-first, minimal-dependency project.
2. **Seed with English + a *generated*, debug-only `en_XA` pseudo-locale; no
   real translation yet** (2026-07-04). The pseudo-locale proves and *enforces*
   the full path (missing keys, overflow, un-localized text) with zero
   translator burden and zero drift, because it is generated from the template.
   It is a **debug/profile-only** selectable locale — filtered out of
   `supportedLocales` in release (`kReleaseMode`) so real users never resolve
   to it — while its `app_en_XA.arb` still ships in the repo. A real language
   is a later drop-in ARB; building the machinery must not wait on translation.
3. **Enforce with both a Dart CLI gate and a `custom_lint` rule**
   (2026-07-04). They catch different moments (commit/CI vs. edit-time) with
   different bypass profiles; the CLI gate is the un-bypassable source of truth
   and does not depend on contributors loading the analyzer plugin, while the
   lint gives the fast in-IDE feedback the gate cannot. The gate is written in
   **Dart, not shell**, so it runs identically on Windows, macOS, and Linux
   with no `grep`/`bash` dependency on the every-commit path — the reason a
   `.sh` gate was rejected. Redundancy is the point.
4. **Generated l10n output is git-ignored, regenerated by the build, and
   excluded from coverage** (2026-07-04). Derived code should not be committed
   or counted; `generate: true` reproduces it on `pub get`/`test`/`build`, and
   it joins `.g.dart` in the single coverage exclusion list. If a toolchain
   step ever needs the files present without a prior build, the fallback is to
   commit them — but that is not expected, since `flutter analyze` runs after
   `pub get` has generated them.
5. **Device-locale resolution now; in-app override deferred behind an
   `AppSettings` seam** (2026-07-04). Shipping a language picker before there
   is a second language is premature; reserving `localeOverride` in
   `AppSettings` (feeding `MaterialApp.locale`) keeps the door open without
   building it, reusing the existing themeability seam so a later picker
   rebuilds content only.
6. **Ship a beginner-oriented "add a language" contributor guide**
   (2026-07-04). Community translation is the entire payoff of i18n, and
   neither the maintainer nor most contributors are i18n experts, so the
   low-barrier path (`gen-l10n` auto-discovers `app_<locale>.arb`, so a
   language is a one-file drop with no Dart change) must be written down, not
   tribal knowledge. It lives at `lib/l10n/README.md`, co-located with the ARBs
   for discoverability, covers both the UI-string and built-in-content tracks,
   and is validated by following it rather than by review alone.
7. **Localize built-in content via a locale-partitioned asset layout with
   per-file `en` fallback; structure now, translations later** (2026-07-04).
   The tutorial and help engrams are user-facing whole documents, not covered
   by ARB, and restructuring four markdown files now is far cheaper than doing
   it once there are dozens. The base locale defines the canonical page set and
   each read falls back to `en` per file, so a partial or lagging translation
   degrades into a mixed-but-whole document rather than a broken one — the
   decisive property for high-volume, always-trailing content. Locale is
   injected into the built-in's `AssetEngramStore` from context where the
   engram is opened, keeping the `EngramStore` contract locale-free (filesystem
   engrams have none). The machinery and tests land now with English-only
   content; translated prose is ongoing. Extends engram-storage Decision 5,
   which assumed one fixed asset prefix per built-in.
